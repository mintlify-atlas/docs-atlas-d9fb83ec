## Service Order Management

Service orders represent client requests and track work from initiation to completion. Each service order automatically creates an associated protocol for detailed work tracking.

<AccordionGroup>
  <Accordion title="Service Order Lifecycle">

## Protocol System

Protocols track the detailed work items, products, and services for each service order or subscription. They manage the complete lifecycle from work assignment through billing.

<Info>
  Protocols are automatically created when a new service order is created. They serve as the central hub for tracking all work items, products, receipts, and billing.
</Info>

### Protocol Components

<Tabs>
  <Tab title="Protocol Registers">
    Track individual work items or services with hourly/fixed pricing:
    
    ```typescript
    // Add work item to protocol
    POST /api/protocols/registers
    {
      "ProtocolId": 15,
      "description": "Frontend development",
      "value": 2500.00,
      "hours": 40
    }
    ```
    
    Work items contribute to the total protocol value and trigger billing calculations.
  </Tab>
  
  <Tab title="Protocol Products">
    Add products or recurring services to protocols:
    
    ```typescript
    // Add product to protocol
    POST /api/protocols/products
    {
      "ProtocolId": 15,
      "ProductId": 3,
      "value": 150.00,
      "charge_type": "Mensal"
    }
    ```
    
    Products can have charge types: `Único`, `Mensal`, or `Anual`. Recurring charges automatically create subscriptions when the protocol is closed.
  </Tab>
  
  <Tab title="Receipts">
    Track payments received against protocols:
    
    ```typescript
    // Record payment receipt
    POST /api/protocols/receipts
    {
      "ProtocolId": 15,
      "value": 1000.00,
      "note": "Partial payment - 50% deposit",
      "BankAccountId": 2
    }
    ```
    
    Receipts reduce the outstanding balance and trigger protocol closure when fully paid.
  </Tab>
</Tabs>

### Protocol Status Flow

```typescript
status: 'Em aberto' | 'Liberado para pagamento' | 'Fechado' | 'Cancelado'
```

<Steps>
  <Step title="Em aberto">
    Initial state. Protocol is being populated with work items and products.
  </Step>
  
  <Step title="Liberado para pagamento">
    Protocol is complete and ready for billing. Automatically creates billing entry and adds to client's next invoice.
    
    ```typescript
    // When protocol status changes to 'Liberado para pagamento'
    const billing = await Billing.create(
        { ClientId: client.id, status: 'pendente', due_date: nextDueDate },
        { transaction }
    )
    
    const value = protocol_products.reduce((sum, v) => sum + v.value, 0) +
                  protocol_registers.reduce((sum, v) => sum + v.value, 0) -
                  receipts.reduce((sum, v) => sum + v.value, 0)
    
    await BillingProtocol.create(
        { BillingId: billing.id, ProtocolId: protocol.id, value },
        { transaction }
    )
    ```

  </Step>
  
  <Step title="Fechado">
    Protocol is complete and fully paid. Service order is automatically marked as Finalizado.
    
    <Warning>
      Protocols cannot be closed until receipts cover the full protocol value:

### Subscription Worker

A background worker runs every 10 minutes to manage subscription lifecycle:

```typescript
private async loop(): Promise<void> {
    const protocols = await Protocol.findAll({
        where: {
            status: ['Em aberto', 'Liberado para pagamento'],
            SubscriptionId: { [Op.gt]: 1 }
        },
        include: [Subscription]
    })
    
    for (let protocol of protocols) {
        // Release for payment 7 days before due date
        if (moment(protocol.Subscription.dueAt).diff(moment(), 'days') > 7) continue
        
        protocol.status = 'Liberado para pagamento'
        protocol.Subscription.status = 'Pendente'
        
        // Mark as unpaid 3 days after due date
        if (moment().diff(moment(protocol.Subscription.dueAt).add(3, 'days'), 'days') >= 0) {
            protocol.Subscription.status = 'Não pago'
        }
    }
}
```

## Authentication & Security

### JWT Authentication

API endpoints are protected using JWT tokens with 1-hour expiration:

```typescript
static async auth(req: authRequest, res: FastifyReply): Promise<FastifyReply> {
    const { username, password } = req.body
    const user = await User.findOne({ where: { username } })
    
    if (!user || !bcrypt.compareSync(password, user.password)) {
        return res.status(401).send({ error: true, message: 'Usuário ou senha invalidos.' })
    }
    
    const token = await jsonwebtoken.sign(
        { user }, 
        String(process.env.SECRET), 
        { expiresIn: '1h' }
    )
    
    return res.send({
        type: 'bearer',
        token,
        expires_in: date.valueOf()
    })
}
```

### Middleware Protection

Protected routes use the `isAuthed` middleware:

```typescript
export default function isAuthed(req: Request, res: Reply, next: DoneFunction): void {
    const { authorization } = req.headers
    
    const decoded = jsonwebtoken.verify(
        authorization?.replace('bearer ', '').replace('Bearer ', '') as string,
        process.env.SECRET as string
    ) as User
    
    req.__user = decoded
    next()
}
```
