## Overview

Protocols are the financial tracking backbone of DevEfficiency. They link service orders and subscriptions to billable items (products and labor), manage payment status, and generate receipts. Every service order and subscription automatically gets a protocol.

## Protocol Structure

Each protocol contains the following key fields:

```typescript
interface Protocol {
  id: number
  status: ProtocolStatus       // Payment/closure status
  notes?: string               // Additional notes
  ServiceOrderId?: number      // Link to service order
  SubscriptionId?: number      // Link to subscription
  closedAt?: Date             // When protocol was closed
  createdAt: Date
  updatedAt: Date
}
```

<Info>
Protocols are automatically created when service orders or subscriptions are created. You typically don't create protocols manually.
</Info>

## Protocol Status

Protocols progress through a payment and closure workflow:

### Status Values

From `src/models/Protocol.ts:72-75`:

```typescript
status: {
  type: DataTypes.ENUM,
  values: ['Em aberto', 'Liberado para pagamento', 'Fechado', 'Cancelado'],
  defaultValue: 'Em aberto',
}
```

<Accordion title="1. Em aberto (Default)">
**Initial Status**: Work is in progress, items are being added.

**Activities:**
- Add protocol products (billable items)
- Add protocol registers (labor/expenses)
- Update protocol notes
- Work is ongoing

**Cannot:**
- Generate billing
- Mark as paid
</Accordion>

<Accordion title="2. Liberado para pagamento">
**Ready for Billing**: Work is complete, ready to bill client.

**Activities:**
- Generate invoices
- Create receipts
- Track payments
- Include in billing PDFs

**Billing Generation:**
Protocols with this status are included in billing reports and invoices.
</Accordion>

<Accordion title="3. Fechado">
**Closed**: Work is complete and paid, protocol is closed.

**Triggers:**
- Service order marked "Finalizado"
- All payments received
- Protocol manually closed

**Effects:**
- `closedAt` timestamp is set
- Included in revenue calculations
- Cannot be modified
</Accordion>

<Accordion title="4. Cancelado">
**Cancelled**: Work was cancelled, no billing.

**Triggers:**
- Service order marked "Cancelado"
- Client cancellation
- Project abandonment

**Effects:**
- Excluded from billing
- Excluded from revenue reports
- Records preserved for history
</Accordion>

## Products and Services

### Protocol Products


**Relationship:**
```typescript

```

### Product Association

Products are linked from the product catalog:

```typescript
{
  "Protocol": {
    "id": 1,
    "Protocol_products": [
      {
        "id": 1,
        "quantity": 5,
        "unit_price": 100.00,
        "total": 500.00,
```

<Warning>
**RESTRICT Delete**: Cannot delete a protocol if it has associated products. Remove products first or change the deletion approach.
</Warning>

## Querying Protocols

Filter protocols by client, status, and pagination:

```typescript
GET /protocols?page=1&limit=10&ClientId=10&status=Liberado para pagamento
```

**Query Parameters:**
- `page` - Page number (default: 1)
- `limit` - Items per page (default: 10, use -1 for all)
- `status` - Comma-separated status values
- `ClientId` - Filter by client (via service orders or subscriptions)

**Response:**
```json
{
  "data": [
    {
      "id": 1,
      "status": "Liberado para pagamento",
      "Protocol_registers": [...],
      "Protocol_products": [...],
      "Receipts": [...],
      "Service_order": {
        "id": 1,
        "subject": "Website Redesign",
        "Client": {...}
      }
    }
  ],
  "total": 15,
  "page": 1,
  "limit": 10
}
```

### Client Filtering

From `src/controllers/protocolsController.ts:48-57`, client filtering works through service orders and subscriptions:

```typescript
if (ClientId) {
```

## PDF Generation

Generate combined protocol PDFs for clients:

```typescript
GET /protocols/pdf?ClientId=10&type=Liberado para pagamento
```

**Parameters:**
- `ClientId` - Required, client to generate PDF for
- `type` - Protocol status to include (default: "Liberado para pagamento")

**PDF Contents:**
From `src/controllers/protocolsController.ts:100-172`:
- All service orders for client with matching protocol status
- All subscriptions for client with matching protocol status
- Protocol products with pricing
- Protocol registers (labor and expenses)
- Receipts and payment information
- Company branding from Config

**Technical Details:**
- Uses `budget_multi.ejs` template
- Puppeteer for PDF generation
- Filename: `protocols_{client_name}_{timestamp}.pdf`

<Note>
**Multi-Protocol PDF**: This endpoint generates a single PDF containing all protocols for a client with the specified status. Useful for monthly billing packages.
</Note>

## Updating Protocols

Update protocol status and notes:

```typescript
// From src/controllers/protocolsController.ts:87-97
PUT /protocols/:id

{
  "status": "Liberado para pagamento",
  "notes": "Ready for March billing cycle"
}
```

**Status Change Effects:**
- **To "Fechado"**: Sets `closedAt` timestamp (via hook)
- **To "Liberado para pagamento"**: Includes in billing generation
- **To "Cancelado"**: Excludes from all billing

### Automatic Updates

Protocols are automatically updated when service orders change:

```typescript
// From src/controllers/serviceOrdersController.ts:99-100
if (status == 'Finalizado') await protocol?.update({ status: 'Fechado' })
if (status == 'Cancelado') await protocol?.update({ status: 'Cancelado' })
```

## Billing Integration

Protocols link to billing through the BillingProtocol join table:

```typescript
// From src/models/Protocol.ts:113-117
Protocol.hasMany(BillingProtocol, {
  onDelete: 'RESTRICT',
})

BillingProtocol.belongsTo(Protocol)
```

**Billing Relationship:**
- Many-to-many through BillingProtocol
- Multiple protocols can be in one billing
- One protocol can be split across multiple billings
- Tracks which protocols are included in each invoice

## Common Workflows

### Service Order Billing

1. **Service Order Created** → Protocol auto-created with status "Em aberto"
2. **Add Products/Services** → Protocol products linked
3. **Track Labor** → Protocol registers added
4. **Work Complete** → Service order "Finalizado", protocol → "Fechado"
5. **Generate Billing** → Protocol included in invoice
6. **Payment Received** → Receipt created

### Subscription Billing

1. **Subscription Created** → Protocol auto-created
2. **Monthly Cycle** → Protocol status → "Liberado para pagamento"
3. **Generate Invoice** → Include all "Liberado" protocols
4. **Payment Received** → Create receipt, close protocol

### Multi-Protocol Invoice

1. **Multiple Protocols Ready** → All have status "Liberado para pagamento"
2. **Generate Combined PDF** → Use `/protocols/pdf?ClientId=X`
3. **Create Billing Record** → Link all protocols to billing
4. **Track Payment** → Update all protocols when paid
